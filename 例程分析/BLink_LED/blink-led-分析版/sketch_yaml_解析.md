# `sketch.yaml` 解析

## 原文件作用

`sketch.yaml` 用来描述 Arduino 侧草图的构建配置，重点是：

- 目标平台；
- 所需依赖库；
- 默认使用哪个构建配置。

这个文件不会直接控制 LED，但它决定了项目能否正确编译，以及编译时要引入哪些库。

## 原始内容解析

### `profiles:`

这里定义的是构建配置集合。一个项目理论上可以有多个 profile，例如：

- 面向不同开发板的构建方式；
- 面向不同依赖版本的构建方式；
- 调试版与发布版的构建方式。

本项目只定义了一个 `default` 配置。

### `default:`

这是默认配置的名字。

后面所有缩进在它下面的内容，都是这个默认构建配置的一部分。

### `platforms:`

这里定义目标编译平台。

项目中写的是：

`arduino:zephyr`

这说明该例程并不是传统意义上只依赖 AVR 风格的经典 Arduino 构建方式，而是面向当前这个支持相应运行环境的平台配置。

从项目整体结构看，这个例程是为了支持 Python 与 Arduino 桥接通信，因此平台配置必须和该运行体系兼容。

### `libraries:`

这里列出编译该草图时需要的库依赖。

#### `Arduino_RouterBridge (0.3.0)`

这是最核心的库。

作用：

- 提供 `Bridge.begin()`；
- 提供 `Bridge.provide(...)`；
- 建立 Python 与 Arduino 之间的桥接调用能力。

没有这个库，就无法实现本例中的远程函数调用。

#### `dependency: Arduino_RPClite (0.2.1)`

这是 Router Bridge 依赖的通信基础组件之一。

从命名可以推测，它用于更底层的轻量级远程过程调用能力支持，是桥接机制能够工作的基础依赖之一。

#### `dependency: ArxContainer (0.7.0)`

这是一个容器类相关依赖库。

通常用于提供更方便的内部数据结构支持，例如动态数组、容器封装等。它不是本例业务逻辑直接使用的 API，但可能被桥接库内部使用。

#### `dependency: ArxTypeTraits (0.3.2)`

这是类型特征相关依赖库。

通常用于模板、类型推断、编译期类型判断等场景，也更偏向底层支撑。

#### `dependency: DebugLog (0.8.4)`

这是调试日志支持库。

在通信、桥接、错误排查等场景中常见，虽然本例代码没有直接调用它，但依赖链中可能需要它提供调试输出能力。

#### `dependency: MsgPack (0.4.2)`

这是一个消息打包/序列化相关依赖。

在桥接通信场景里，经常需要把参数和调用请求进行编码与传输。这个库很可能承担消息序列化或反序列化职责。

### `default_profile: default`

这表示项目默认采用名为 `default` 的构建配置。

也就是说，如果没有额外指定 profile，系统就会用前面定义的默认配置来编译这个草图。

## 小结

`sketch.yaml` 是“编译与依赖配置文件”，核心职责包括：

- 指定草图编译目标平台；
- 声明 Router Bridge 及其相关依赖；
- 指定默认构建配置。

它的重点不是“程序怎么闪灯”，而是“程序如何被正确构建并具备桥接通信能力”。
