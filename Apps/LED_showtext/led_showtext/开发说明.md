# 开发说明

本文档面向后续维护和扩展 `LED_showtext` 工程时使用，重点说明当前架构、关键扩展点以及修改建议。

## 1. 当前架构

这个 App 使用 UNO Q 的双端模型：

- `python/main.py`
  运行在 Linux / MPU 侧，负责读取文本输入源并调用 Bridge

- `sketch/sketch.ino`
  运行在 MCU 侧，负责 LED 矩阵显示

- `Arduino_RouterBridge`
  负责在两端之间传递文本

当前数据流：

```text
python/led_text.txt
    -> python/main.py
    -> Bridge.call("set_text", text)
    -> sketch.setText(...)
    -> renderScrollingText(...)
    -> UNO Q LED matrix
```

## 2. 为什么这样设计

这种设计把“输入来源”和“显示实现”分开了：

- Python 端只负责拿到新文本并通知 MCU
- MCU 端只负责显示

这样做的好处：

- 输入来源可以替换
- 显示逻辑集中在一处
- 后续更容易接入 Web UI、传感器、按钮或网络接口

## 3. 当前核心逻辑

### Python 侧

`python/main.py` 的核心函数：

- `read_text_file()`
  负责从 `led_text.txt` 读取内容

- `send_text_if_changed()`
  负责比较内容是否变化，只有变化后才调用 `Bridge.call(...)`

- `loop()`
  每隔 `POLL_INTERVAL_SEC` 秒执行一次轮询

当前策略是“轮询文件变更”，而不是监听文件系统事件。  
这样实现简单、稳定，也更适合 App Lab 的运行环境。

### Sketch 侧

`sketch/sketch.ino` 的核心函数：

- `renderScrollingText(const String& text)`
  用 `ArduinoGraphics` 配合 `Arduino_LED_Matrix` 绘制滚动文字

- `setText(String text)`
  作为 RouterBridge 的 RPC 接口，接收 Python 发送的新文本

- `loop()`
  持续重播当前滚动动画，使文本循环显示

## 4. 为什么 `loop()` 里要重复调用渲染

如果只在 `setup()` 或 `setText()` 里调用一次滚动逻辑，文本通常只会滚动一轮。  
为了实现“持续循环滚动”，当前做法是在 `loop()` 中不断重新播放当前文本。

如果后续你发现：

- 循环之间有明显停顿
- 文本重播不够平滑

那么优化方向应该从 `renderScrollingText()` 的调用策略入手，而不是先改 Python 侧。

## 5. 适合扩展的位置

## 5.1 更换输入源

当前输入源是：

- `python/led_text.txt`

后续可以替换成：

- Web UI 表单输入
- Python 端固定变量
- 网络 API 返回内容
- 串口或 Socket 输入
- 按钮触发的预设文本

扩展时主要改：

- `python/main.py`

而 `sketch/sketch.ino` 可以保持不变，只要仍然调用：

```python
Bridge.call("set_text", text)
```

## 5.2 增加字符过滤

UNO Q 的 LED 矩阵尺寸很小，`Font_5x7` 对复杂字符支持有限。  
如果后续要增强稳定性，建议在 Python 侧加一层过滤。

建议位置：

- `python/main.py` 中，在 `Bridge.call(...)` 前增加文本清洗函数

典型处理方式：

- 去掉前后空白
- 限制最大长度
- 过滤不支持的字符
- 将连续空格压缩为单个空格

## 5.3 增加多种显示模式

现在只有一种模式：

- 左滚动文本

后续可以扩展成：

- 静态显示短文本
- 左滚动
- 右滚动
- 上下滚动
- 文本前加图标
- 文本闪烁

适合修改的位置：

- `sketch/sketch.ino`

建议做法：

- 保留 `setText(...)`
- 新增 `setMode(...)`
- 在 MCU 侧维护一个 `currentMode`

这样 Python 端就可以同时调用：

```python
Bridge.call("set_text", text)
Bridge.call("set_mode", "scroll_left")
```

## 5.4 接 Web UI

如果后续你想做浏览器输入文本后立刻显示到 LED 矩阵，建议架构是：

```text
Web UI
  -> python/main.py
  -> Bridge.call("set_text", text)
  -> sketch/sketch.ino
```

也就是说：

- Web UI 只和 Python 通信
- Python 再把文本下发给 MCU

不建议让 Web UI 直接控制每个像素，因为这样会把显示逻辑打散。

## 5.5 接实体按钮

如果你想通过硬件按钮切换预设文本，有两种思路：

### 方案 A：按钮接 MCU

优点：

- 响应快
- 结构简单

做法：

- 在 `sketch.ino` 里直接读取按钮
- 按键后修改 `currentText`

### 方案 B：按钮接 Python 侧逻辑

如果按钮事件先进入 Python，则保持当前架构即可：

- Python 处理事件
- 再调用 `Bridge.call("set_text", text)`

## 6. 当前参数说明

### Python 侧

- `TEXT_FILE`
  当前文本文件路径

- `POLL_INTERVAL_SEC`
  文件轮询周期，默认 `0.5`

### Sketch 侧

- `Font_5x7`
  当前字体

- `matrix.textScrollSpeed(70)`
  滚动速度

- `SCROLL_LEFT`
  当前滚动方向

如果要调效果，优先改这几个参数。

## 7. 修改建议

推荐的修改顺序：

1. 先改 `python/led_text.txt` 验证显示链路
2. 再改 `sketch/sketch.ino` 的速度、字体和滚动方向
3. 最后再引入新的输入源或新的显示模式

这样更容易定位问题。

## 8. 常见问题

### 8.1 App 启动但不显示文本

优先检查：

- `python/led_text.txt` 是否为空
- Python App 是否正常运行
- `Bridge.call("set_text", text)` 是否成功

### 8.2 文本只滚动一次

说明 `loop()` 中没有持续重播当前文本。  
当前工程已经通过在 `loop()` 中重复调用渲染解决了这个问题。

### 8.3 修改文件后显示没有更新

优先检查：

- 文件是否已保存
- `POLL_INTERVAL_SEC` 是否太长
- 新文本是否和旧文本完全相同

### 8.4 文本显示不清楚

原因通常有三个：

- 矩阵尺寸太小
- 字体不适合
- 字符本身太复杂

这时建议：

- 尽量使用短文本
- 使用英文、数字、简单符号
- 保持 `Font_5x7`

## 9. 建议的下一步扩展

如果要继续把这个工程做成更完整的应用，优先级建议如下：

1. 在 Python 侧加入文本过滤和长度限制
2. 增加多种显示模式
3. 接入 Web UI
4. 增加图标和文本组合显示
5. 加入按钮或网络输入源

## 10. 结论

当前工程已经具备一个清晰、稳定的最小可扩展结构：

- Python 侧负责接收或生成文本
- Bridge 负责传输
- MCU 侧负责最终显示

后续无论是接文件、网页、按钮还是网络接口，都建议保持这个分层方式，不要把所有逻辑堆到单一一端。
