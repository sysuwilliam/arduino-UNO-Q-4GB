## 1. 这是标准的双端 App 结构

官方文档明确说明，UNO Q 的 App 常见结构是：

- `main.py` 运行在 Linux 系统侧；
- `sketch.ino` 运行在微控制器侧；
- 两者通过 Bridge 通信。

这说明 Blink 例程虽然功能很小，但工程模型并不简单，它已经完整体现了 UNO Q App 的典型结构。

## 2. `app.yaml` 是平台管理的元数据文件

官方入门指南指出，`app.yaml` 用于保存应用名称、描述和 Bricks 等元数据，并标注为不可随意编辑的文件。

对于这个 Blink 例程来说，需要特别注意：

- `bricks: []` 表示这个 App 没有使用额外 Brick；
- 不应通过手改 `app.yaml` 来给项目“补功能”；
- 真正业务逻辑仍然在 `main.py` 和 `sketch.ino` 中。

## 3. `App.run()` 必须放在 `main.py` 末尾

官方文档特别强调，`App.run()` 会启动 App 运行时、Bridge，以及所有已导入的 Bricks 或工具。

因此这个例程中的：

```python
App.run(user_loop=loop)
```

不仅是“开始循环”，更是整个 App 生命周期的启动入口。

## 4. Bridge 不只是一个函数调用封装

官方把 Bridge 的能力明确分成三类：

- `Provide`
- `Call`
- `Notify`

Blink 例程只展示了其中最经典的一种组合：

- Arduino 侧 `Bridge.provide(...)`
- Python 侧 `Bridge.call(...)`

这正是最适合初学者理解的最小闭环。

## 5. 日志调试方式可以补充理解

官方文档说明：

- Python 端 `print()` 输出会出现在 App Lab 的 `Main (Python)` 日志标签中；
- sketch 端若要显示在 App Lab Console，应使用 `Monitor.print()`，而不是 `Serial.println()`。

因此在后续扩展这个示例时：

- 若想调试 Python 定时逻辑，可以直接用 `print()`；
- 若想调试 sketch 侧状态，不应默认依赖 `Serial.println()`。

## 6. 官方示例概览中的定位

官方示例总览对 Blink 的定位非常明确：

- 它是“无需额外硬件”的基础示例；
- 核心目标是展示 LED 控制与 Router Bridge 通信。

因此它最适合被当作：

- UNO Q App 结构入门；
- Python/MCU 双端协作入门；
- Router Bridge 基础调用入门。
