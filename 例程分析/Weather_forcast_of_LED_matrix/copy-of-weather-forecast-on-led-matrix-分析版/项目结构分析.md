# 项目结构分析

## 目录结构

原始项目的核心结构如下：

```text
copy-of-weather-forecast-on-led-matrix/
├── README.md
├── app.yaml
├── python/
│   └── main.py
└── sketch/
    ├── sketch.ino
    ├── sketch.yaml
    └── weather_frames.h
```

各文件职责如下：

- `README.md`
  官方说明文档，介绍用途、运行方式和基本原理。

- `app.yaml`
  App Lab 项目元数据文件，声明名称、说明、图标和 Brick 依赖。根据官方文档，这类元数据由平台管理，不应作为普通业务源码手动维护。

- `python/main.py`
  Python 端逻辑，负责根据城市名称获取天气分类。

- `sketch/sketch.ino`
  Arduino 端逻辑，负责请求天气并在 LED 点阵上播放对应动画。

- `sketch/sketch.yaml`
  Arduino 侧构建与依赖配置文件。

- `sketch/weather_frames.h`
  天气动画帧资源文件。

## 架构分层

这个项目体现出比普通闪灯例程更完整的四层结构：

### 1. 数据获取层

由 Python 和 `weather_forecast` Brick 负责。

作用：

- 根据城市名访问天气服务；
- 获取天气结果；
- 将原始天气信息整理为较简单的分类。

结合官方 Bricks 文档，可以进一步理解成：

- `main.py` 运行在 UNO Q 的 Linux 侧；
- `weather_forecast` Brick 也工作在 Linux 侧的 App 运行环境中；
- Python 主程序通过 Brick API 获取结果，而不是直接在 sketch 中访问网络。

### 2. 桥接通信层

由 Router Bridge 负责。

作用：

- Arduino 把城市名传给 Python；
- Python 把天气分类结果返回给 Arduino。

这里使用的是 Bridge 的“请求-响应”模式。官方还定义了 `Notify` 这种单向推送模式，但本项目没有使用。

### 3. 显示控制层

由 `sketch.ino` 负责。

作用：

- 根据返回结果选择动画；
- 控制动画播放次数；
- 组织 LED 点阵显示流程。

官方示例概览把这个项目归类为“天气数据由 Brick 获取，LED 显示由 Arduino sketch 管理”，这与当前源码结构完全吻合。

### 4. 动画资源层

由 `weather_frames.h` 负责。

作用：

- 提供不同天气类型的点阵动画帧；
- 让显示层按需加载和播放。

## 运行流程

程序运行时的流程可以概括为：

1. Arduino 初始化 LED 点阵。
2. Arduino 初始化 Router Bridge。
3. Python 运行时启动并注册 `get_weather_forecast`。
4. Arduino 在 `loop()` 中使用城市名发起桥接调用。
5. Python 收到城市名后查询天气服务。
6. Python 返回简化后的天气分类字符串。
7. Arduino 根据字符串选择动画序列。
8. LED 点阵重复播放相应动画。
9. 下一轮 `loop()` 再次查询并更新显示。

需要补充一个细节：

- 官方 README 常说“每 10 秒更新一次”；
- 但当前源码中没有单独写死 10 秒轮询定时器；
- 实际刷新节奏主要由动画帧时长和 `playRepeat()` 重复次数共同决定。

例如：

- `sunny` 为 3 帧，每帧 500ms，重复 10 次，约 15 秒；
- `rainy` 为 3 帧，每帧 200ms，重复 20 次，约 12 秒；
- `foggy` 为 3 帧，每帧 660ms，重复 5 次，约 9.9 秒。

因此更准确的说法是：它会按“当前动画播放总时长”作为近似刷新周期，而不是所有天气都严格固定 10 秒。

## 调用链分析

从调用关系看，核心链路如下：

```text
Arduino loop()
-> Bridge.call("get_weather_forecast", city)
-> Python get_weather_forecast(city)
-> WeatherForecast().get_forecast_by_city(city)
-> 返回 category
-> Arduino 判断天气类别
-> matrix.loadSequence(...)
-> matrix.playSequence()
```

## 这个项目比 Blink 示例多了什么

和简单闪灯示例相比，这个项目多出了几个关键能力：

- 需要联网获取外部数据；
- 需要处理输入参数 `city`；
- 需要接收返回值；
- 需要根据结果做条件分支；
- 需要调用点阵动画资源进行可视化。

这意味着它已经从“最基础硬件控制”进入了“数据驱动显示应用”的层次。

## 设计优点

- 结构分层清晰；
- 逻辑和动画资源分离；
- Python 擅长联网，Arduino 擅长硬件显示，分工合理；
- 天气类别与显示方式解耦，便于后续扩展更多天气动画。

## 设计局限

- `loop()` 中没有显式延时或更新周期控制，更新节奏依赖动画播放时长；
- 没有处理未知天气类别；
- 没有处理网络失败后的降级显示；
- 城市名写死在 Arduino 代码中，灵活性一般；
- 只展示分类结果，没有展示温度、湿度等更多信息。

## 可扩展方向

- 支持更多天气类型；
- 在点阵上显示温度或滚动文字；
- 增加失败提示动画；
- 把城市设置改成可通过 Python、串口或界面动态修改；
- 缓存天气结果，减少重复请求；
- 让更新周期更可控，例如固定每 10 分钟更新一次天气。
