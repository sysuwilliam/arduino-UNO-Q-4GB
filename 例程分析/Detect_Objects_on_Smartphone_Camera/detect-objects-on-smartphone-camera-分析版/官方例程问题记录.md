# 官方例程问题记录

本文档用于记录在阅读官方示例 `Detect Objects on Smartphone Camera` 时发现的代码问题，方便后续分析、讲解和扩展时参考。

## 1. 置信度输入框上限判断错误

### 问题位置

文件：

- `assets/app.js`

具体函数：

- `handleConfidenceInputChange()`
- `validateConfidenceInput()`

### 官方原始代码

原始逻辑中包含以下判断：

```javascript
if (value > 1) value = 100;
```

### 为什么这里有问题

这个页面里的置信度控制实际上采用的是 **0 到 100 的百分比输入模型**，而不是 0 到 1。

证据有三处：

#### 1. 滑块范围定义是 0 到 100

```html
<input type="range" id="confidenceSlider" min="0" max="100" step="1" value="50">
```

#### 2. 页面显示也是百分比值

```javascript
confidenceValueDisplay.textContent = displayValue;
```

#### 3. 真正发给后端时才转换成 0 到 1

```javascript
socket.emit('override_th', value / 100);
```

这说明：

- 前端输入层使用的是 `0 ~ 100`
- 后端模型层需要的是 `0.0 ~ 1.0`
- 两者之间在发送时通过 `/ 100` 做转换

因此，输入框的上限判断如果写成：

```javascript
if (value > 1) value = 100;
```

就和整个页面的数据模型不一致。

### 实际影响

这个错误不是“程序完全无法运行”的致命问题，但会直接影响前端交互体验。

例如：

1. 用户在输入框里输入 `50`
2. 程序判断 `50 > 1`
3. 于是直接把值改成 `100`

结果就是：

- 用户手动输入 `50`，却会被意外重置为 `100`

这会导致：

- 输入框行为异常
- 用户难以精确输入检测阈值
- 页面逻辑和视觉设计不一致

### 为什么不是致命错误

它不是致命错误，原因是：

- 如果用户只拖动滑块，而不手动输入数值，这个问题不一定明显暴露
- 后端 AI 检测功能本身不会因此完全失效
- App 的主流程仍然可以工作

所以更准确的定性应该是：

- **不是致命错误**
- **但属于明确的前端交互逻辑 bug**

### 正确写法

正确的上限判断应为：

```javascript
if (value > 100) value = 100;
```

### 修正后的逻辑

在本地分析和修正版中，已将这两处判断修正为：

```javascript
if (value > 100) value = 100;
```

### 结论

这个问题说明一个很重要的工程经验：

- 即使是官方例程，也可能存在局部逻辑错误
- 阅读代码时仍然需要结合上下文验证数据模型是否一致

在这个示例中，问题的根源不是算法，也不是 AI Brick，而是：

- **前端输入层和阈值数据模型之间的边界判断写错了**
