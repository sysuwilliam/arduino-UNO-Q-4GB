# 项目结构分析

## 目录结构

原始项目的核心结构如下：

```text
copy-of-blink-led/
├── README.md
├── app.yaml
├── python/
│   └── main.py
└── sketch/
    ├── sketch.ino
    └── sketch.yaml
```

各部分职责如下：

- `README.md`
  项目的官方说明文档，介绍用途、运行方式和工作原理。

- `app.yaml`
  App Lab 层面的应用元数据配置文件。根据官方文档，它属于平台管理文件，不应作为普通源码手动编辑。

- `python/main.py`
  Python 侧的业务逻辑代码，负责计时和状态切换。

- `sketch/sketch.ino`
  Arduino 侧的硬件控制代码，负责驱动板载 LED。

- `sketch/sketch.yaml`
  Arduino 草图编译与依赖配置文件。

## 架构划分

这个例程虽然简单，但结构上体现了明确的分层设计：

### 1. Python 层

作用：

- 决定业务节奏；
- 保存 LED 当前状态；
- 发起桥接调用。

特点：

- 更适合写高层逻辑；
- 代码表达更紧凑；
- 便于快速调整时间、状态逻辑和业务流程。

官方文档还强调，`main.py` 运行在 UNO Q 的 Linux 系统一侧，而不是微控制器一侧。

### 2. Bridge 桥接层

作用：

- 连接 Python 与 Arduino；
- 把 Python 的函数调用请求转发给 Arduino；
- 负责跨运行环境的数据传递。

特点：

- 是本项目最关键的中间层；
- 让高层逻辑和底层硬件控制解耦。

根据官方入门指南，Bridge 的核心能力不只包括 `Call`，还包括：

- `Provide`：暴露一个可被另一端调用的服务；
- `Call`：调用另一端已经暴露的服务；
- `Notify`：单向推送数据。

本例只使用了其中最基础的 `Provide + Call` 模式。

### 3. Arduino 硬件层

作用：

- 初始化引脚；
- 接收 Python 发来的状态参数；
- 对 GPIO 进行实际写操作。

特点：

- 紧贴硬件；
- 负责最终执行；
- 逻辑简单但必须准确。

这里的 `sketch.ino` 运行在 UNO Q 的微控制器一侧，因此适合承担 GPIO 控制这类底层硬件任务。

## 运行流程

项目运行时，整体流程如下：

1. Arduino 启动，执行 `setup()`。
2. 初始化板载 LED 引脚为输出模式。
3. 初始化 Router Bridge。
4. 注册 `set_led_state` 供 Python 远程调用。
5. Python 应用启动，持续执行 `loop()`。
6. 每隔 1 秒切换一次 `led_state`。
7. Python 调用 `Bridge.call("set_led_state", led_state)`。
8. Arduino 接收到调用后执行 `set_led_state(bool state)`。
9. `digitalWrite()` 修改板载 LED 对应引脚电平。
10. LED 实现亮灭交替。

## 调用链分析

从函数调用角度看，可以整理成下面这条链路：

```text
App.run(user_loop=loop)
-> Python loop()
-> Bridge.call("set_led_state", led_state)
-> Arduino Bridge.provide("set_led_state", set_led_state)
-> set_led_state(bool state)
-> digitalWrite(LED_BUILTIN, ...)
```

这条链路说明：

- Python 不直接访问硬件；
- Arduino 不主动决定闪烁时机；
- 两端通过桥接协议协作完成完整功能。

从官方 App 运行模型看，每次启动这个例程时：

- App Lab 会先编译 `sketch.ino`；
- 再部署并启动 Python 端应用；
- 然后 Bridge 工具开始工作。

因此这个例程虽然看起来很小，但本质上仍然是一个“Linux 侧 Python + MCU 侧 sketch”的双端应用。

## 设计特点

### 优点

- 分层清晰，容易理解；
- 便于把“业务逻辑”和“硬件控制”分开；
- 非常适合作为 Router Bridge 入门例程；
- 后续可以很容易扩展为按钮控制、串口命令控制、网络控制等更复杂应用。

### 局限

- 功能非常简单，只适合教学或验证环境；
- Python 端使用阻塞式 `sleep()`，不适合复杂并发任务；
- Arduino 端 `loop()` 为空，不展示本地自主逻辑；
- 只演示了单个布尔参数的桥接调用，没有展示复杂数据结构传输。

## 适合继续扩展的方向

- 调整闪烁周期，变成可配置参数；
- 同时控制多个 LED；
- 从 Python 端读取用户输入后再决定 LED 行为；
- 让 Arduino 端把传感器数据回传给 Python；
- 增加异常处理和调试输出；
- 把单个远程函数扩展成一个更完整的硬件控制接口集。
